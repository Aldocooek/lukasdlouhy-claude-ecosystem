=== CASE perf-002 ===

| # | Finding | File:Line | Impact (H/M/L) | Effort (H/M/L) | Fix |
|---|---------|-----------|----------------|----------------|-----|
| 1 | LCP 6.1s — POOR (threshold <2.5s). LCP element is an H1 text node. On a full client-side rendered SPA, the H1 does not exist in the initial HTML — the browser receives a blank document and must download, parse, and execute 1.1MB of JS before the H1 renders. This is a hydration/parsing cost, not layout thrash. | `index.html` (empty shell) + main JS bundle entry point | H | H | Introduce SSR or static pre-rendering (Next.js, Astro, or Gatsby). The H1 must be present in the initial HTML response so the browser can paint it without JS execution. No client-side rendering fix will materially move a text-node LCP on a pure SPA — the architecture must change. |
| 2 | FCP 4.8s — POOR (threshold ideally <1.8s). Three render-blocking third-party analytics scripts in `<head>` without `async` or `defer` block HTML parsing. Each script must fully download and execute before the browser can continue. | `index.html` `<head>` — three `<script src="...analytics...">` tags | H | L | Add `async` or `defer` to all three analytics `<script>` tags. Use `defer` if script execution order matters; `async` if scripts are independent. Expected FCP improvement: 0.8–2s depending on script sizes and network. This is a one-line change per script tag. |
| 3 | Google Fonts loaded via CSS `@import` — render-blocking. The browser must download the CSS file, parse the `@import`, then make a second network request for the font file before rendering can proceed. This is one of the most well-documented FCP killers. | CSS file containing `@import url('https://fonts.googleapis.com/...')` | H | L | Remove the `@import`. Replace with `<link rel="preconnect" href="https://fonts.googleapis.com">` + `<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>` + `<link rel="stylesheet" href="https://fonts.googleapis.com/css2?family=...">` in `<head>`. Alternatively, self-host the font files and preload the WOFF2. |
| 4 | JS bundle 1.1MB uncompressed, no code splitting. The entire application — including routes, components, and dependencies never needed on the blog index — loads before any interaction. This directly causes both the high LCP (text not rendered) and elevated INP 210ms (main thread occupied during parse). | Webpack/Vite config — no `splitChunks` or dynamic imports in use | H | H | Implement route-based code splitting. In React Router: use `React.lazy()` + `Suspense` for each route. In Vite: dynamic `import()` is automatic with route splitting. Target: reduce initial bundle to under 200kb. Third-party deps (analytics, charting, form libs) should be in separate async chunks. |
| 5 | INP 210ms — NEEDS IMPROVEMENT (threshold <200ms; 210ms is borderline). This is not layout thrash — no read/write interleaving is described. The cost is main-thread parsing of the 1.1MB bundle occupying the thread during early interaction. CLS 0.04 — PASS. | Bundle entry point | M | H | INP will improve as a downstream effect of fix 4 (bundle splitting). No separate INP-specific fix required. Validate post-deploy. |

**Biggest bottleneck:** The full client-side rendering architecture is the root cause of every top-three metric failure. The LCP H1 text node cannot be improved within a pure SPA model — a browser that receives an empty HTML shell must execute JavaScript before any meaningful content can paint, which structurally prevents a good LCP regardless of CDN, caching, or network optimizations. Fixes 2 and 3 (async scripts, font loading) are low-effort wins deployable today and will recover 1–2 seconds of FCP. But LCP and the bundle size problem require an architectural decision: introduce SSR, pre-rendering, or at minimum static HTML generation for the blog index. That is the one fix that moves the needle from POOR to GOOD on the metric that matters most.
